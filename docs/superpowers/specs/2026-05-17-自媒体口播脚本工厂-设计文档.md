# 自媒体口播脚本自动生成工厂 — 设计文档

日期：2026-05-17

---

## 1. 概述

短视频MCN机构每天产出大量口播脚本，纯人工产量低、风格一致性差。本项目构建一个多Agent协作流水线，自动化完成：策划→写作→审核→修改的完整创作流程。

### 1.1 核心指标

- 自动审核一次通过率 ≥ 60%
- 多Agent可用率 > 单Agent可用率
- 合规检查拦截率 = 100%（极限词一个不漏）
- 完整流水线耗时 < 60s

---

## 2. 技术决策

| 决策点 | 选型 | 原因 |
|--------|------|------|
| 编排框架 | 自定义状态机 | 线性流程（策划→写作→审核←→修改），while循环足够，无需LangGraph |
| Agent内部 | LangChain | prompt模板 + Pydantic输出解析 |
| LLM | DeepSeek | deepseek-reasoner 策划/审核；deepseek-chat 写作/修改 |
| MCP搜索 | Tavily | AI搜索API，免费额度开发够用，有现成MCP Server |
| 可观测 | LangFuse | 开源免费，一行callback |
| 依赖管理 | pyproject.toml + venv | 现代Python标准 |
| 后端 | FastAPI | 标准异步Web框架 |
| 前端 | Vue 3 + Vite | 简洁易上手，组件文件结构清晰 |
| 前后端通信 | JSON over HTTP | 前端调后端 `/generate` 接口 |

### 2.2 为什么不用 LangGraph

本项目Agent流程是线性的：策划→(确认)→写作→审核→(修改→审核)×N→输出。唯一的分支是 `auto_mode` 开关和审核结果判断。没有并行节点、不需要长期checkpoint、不需要流式中间输出。while + dict 50行代码搞定，LangGraph 会引入额外复杂度和15+依赖。

## 4. 架构设计

### 4.1 流水线状态机

```
输入(选题+风格+时长)
  │
  ▼
策划Agent ──→ 大纲+金句
  │
  ▼
[确认点]  ← auto_mode=false 时暂停，等待人工确认大纲
  │
  ▼
写作Agent ──→ 口播稿
  │
  ▼
审核Agent ──→ issues + 评分 ──┬── passed ──→ 输出终稿
                              │
                              ├── failed + revision_count<3 ──→ 修改Agent
                              │                                    │
                              └── failed + revision_count≥3       │
                                  │                               │
                                  ▼                               │
                              降级输出: 保留当前稿+问题清单 ◀─────┘
                              needs_human=True
```

### 4.2 Agent 接口约定

```python
class BaseAgent:
    """所有 Agent 统一接口"""
    def run(self, state: dict) -> dict:
        """输入 state dict，返回更新后的 state dict."""
        raise NotImplementedError
```

- 每个 Agent 只做三件事：读 state → 调 LLM → 写回 state
- Agent 不知道前一个Agent是谁，也不知道下一步去哪
- 编排器负责 step 流转，Agent 只负责自己的业务逻辑

### 4.3 State 定义

```python
state = {
    "step":            str,       # plan|wait_confirm|write|review|revise|done
    "topic":           str,       # 选题
    "style":           str,       # 账号风格: 搞笑/知识/情感
    "duration":        int,       # 目标时长(秒)
    "auto_mode":       bool,      # True=全自动, False=策划后等人确认
    "confirm_action":  str,       # 确认动作: continue|replan|revise_outline
    "feedback":        str,       # 用户修改/重策划的反馈意见
    "outline":         dict|None, # 策划Agent产出
    "script":          dict|None, # 写作Agent产出
    "review":          dict|None, # 审核Agent产出
    "revision_count":  int,       # 修改次数
    "final_script":    dict|None, # 终稿
    "needs_human":     bool,      # 超限=false, 标记人工
    "grade":           str,       # normal|degraded
    "unresolved_issues": list,   # 未解决问题清单
}
```

## 5. 组件设计

### 5.1 Schemas (Agent间消息)

```python
class Section(BaseModel):
    heading: str
    talking_points: list[str]

class Outline(BaseModel):
    title: str
    hook: str                    # 开头钩子
    sections: list[Section]
    key_phrases: list[str]       # 金句
    estimated_duration: int

class Script(BaseModel):
    content: str                 # 口播稿全文
    duration_estimate: int       # 预估时长(秒)
    tone_markers: list[str]      # 节奏标记 (快/慢/重音)

class Issue(BaseModel):
    severity: str                # P0=合规 P1=事实 P2=风格
    location: str                # 出问题的原文
    description: str             # 问题描述
    suggestion: str              # 修改建议

class ReviewResult(BaseModel):
    passed: bool
    issues: list[Issue]
    score: float                 # 0-100
    checks: dict                 # {compliance: bool, facts: bool, style: bool}
```

### 5.2 Agents

#### 策划Agent (planning.py)
- **职责**: 分析选题 → 搜索热点 → 输出大纲+金句
- **LLM**: deepseek-reasoner（推理强）
- **工具**: Tavily 搜索 (查热点素材)
- **Prompt关键**: 角色设定(MCN资深编导) + 大纲格式约束 + 金句数量要求(≥3条)

#### 写作Agent (writing.py)
- **职责**: 读大纲+风格+时长 → 按口播节奏写稿
- **LLM**: deepseek-chat（快）
- **Prompt关键**: 风格参数输入 + 口播节奏要求(停顿/重音标记) + 时长控制(120字≈1分钟)

#### 审核Agent (review.py)
- **职责**: 读脚本 → 四维审核 → 输出issues+评分
- **LLM**: deepseek-reasoner（分析细致）
- **审核维度**:
  - 信息量(25分): 是否有实质性内容
  - 口语化(25分): 能否读出嘴
  - 合规性(25分): 有无极限词/敏感内容
  - 可用率(25分): 能否直接拿去拍
- **Issue分级**: P0(合规,必改) / P1(事实,必改) / P2(风格,仅记录)
- **工具**: Tavily 搜索 (事实核查) + guardrails.compliance (极限词检测)

#### 修改Agent (revision.py)
- **职责**: 读脚本+issues → 逐条修改P0/P1 → 输出新脚本
- **LLM**: deepseek-chat（快）
- **Prompt关键**: 只改P0/P1,不改P2, 保留原文风格

### 5.3 编排器 (orchestator.py)

```python
MAX_REVISIONS = 3

def run_pipeline(state):
    while state["step"] != "done":
        step = state["step"]

        if step == "plan":
            state["outline"] = planning_agent.run(state)
            state["step"] = "wait_confirm" if not state["auto_mode"] else "write"

        elif step == "wait_confirm":
            # 暂停，等待用户确认。前端再次 POST /generate 传入 confirm_action
            if state["confirm_action"] == "continue":
                state["step"] = "write"
            elif state["confirm_action"] == "replan":
                state["step"] = "plan"        # 重新策划（可选加载 feedback）
                state["outline"] = None
            elif state["confirm_action"] == "revise_outline":
                state["step"] = "plan"        # 调策划Agent改大纲（加载 feedback）
            break  # break 确保一次请求只执行一步，避免 while 循环一次跑完

        elif step == "write":
            state["script"] = writing_agent.run(state)
            state["step"] = "review"

        elif step == "review":
            review = review_agent.run(state)
            state["review"] = review
            must_fix = [i for i in review.issues if i.severity in ("P0", "P1")]

            if not must_fix:
                state["final_script"] = state["script"]
                state["step"] = "done"
            elif state["revision_count"] >= MAX_REVISIONS:
                state["final_script"] = state["script"]
                state["unresolved_issues"] = must_fix
                state["needs_human"] = True
                state["grade"] = "degraded"
                state["step"] = "done"
            else:
                state["step"] = "revise"

        elif step == "revise":
            state["script"] = revision_agent.run(state)
            state["revision_count"] += 1
            state["step"] = "review"

    return state
```

### 5.4 护栏 (compliance.py)

- **极限词检测**: 正则匹配"最好""第一""国家级""唯一""顶级"等广告法禁词
- **医疗敏感审核**: 检测医疗健康类断言（"治愈""根治""神药"等）
- **政治敏感过滤**: 敏感词库匹配
- **格式校验**: 使用 Pydantic 校验 Agent 间消息格式

### 5.5 观测 (langfuse.py)

- 每个Agent: 耗时、Token消耗、重试次数
- 整条流水线: 总耗时
- 审核通过率: 一次通过 vs 需要修改
- 修改轮次分布: 1轮/2轮/3轮/降级
- 层1-3用print打日志，层4用LangFuse callback一键挂上

### 5.6 API (router.py)

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /generate | 发起生成。首次调用传入 topic+style+duration；确认调用传入 state+confirmed=true |
| GET | /health | 健康检查 |

**确认流程**：auto_mode=false 时，首次 POST /generate → 策划Agent跑完返回state（含outline+step=wait_confirm）→ 用户审阅大纲 → 二次 POST /generate 传入同一state+confirmed=true → 编排器从wait_confirm继续执行。

#### 数据流

1. InputForm 收集 topic/style/duration/auto_mode → POST /generate
2. auto_mode=true → 一次跑完返回终稿
3. auto_mode=false → 策划完暂停 → 展示大纲 + 三个按钮：
   - "确认大纲，继续" → confirm_action=continue → 流水线继续
   - "重新策划" → confirm_action=replan + 可选feedback → 重跑策划Agent
   - "修改要求" → confirm_action=revise_outline + feedback → 策划Agent改大纲
4. PipelineView 根据 state.step 更新进度（active/done/idle 三种状态）
5. ScriptOutput 展示 script + review.score + grade + unresolved_issues

## 7. 评估方案

- **数据集**: 10选题×3风格(搞笑/知识/情感) = 30个样本
- **对比**: 每个选题生成2版 — 单Agent直接写 vs 多Agent流水线
- **评分**: 审核Agent对两版分别打分(0-100, 4维度)
- **验证**: 按风格分组算均值，多Agent应稳定高于单Agent
- **人工**: 抽6篇（差最大3篇+差最小3篇）读稿验证评分合理性
